GitHub 上放的是项目，但我希望这里放的是作品。

对我来说，一个仓库如果值得被认真展示，它不应该只剩下名字、语言占比和最后一次提交时间。更重要的是，它为什么会被做出来，我在里面承担了什么，它卡住我的地方在哪里，它最后又把我推向了什么方向。

下面这页不是机械的 repo 清单，而是我给自己公开作品做的一次整理。

## 01. MSCRED-NASA-Anomaly-Detection

### 为什么做

这是我比较明确会放进作品区的项目之一，因为它对应的是我对异常检测、时序建模以及真实数据问题的长期兴趣。比起把它当成一个“深度学习作业”，我更愿意把它理解为一次认真处理复杂信号的尝试：面对多变量、异构、带噪声的数据，模型究竟怎样才能更稳定地看见异常。

### 我做了什么

- 围绕 MSCRED 思路处理 NASA 场景下的异常检测问题
- 关注时空特征的联合建模，而不是只做单一维度判断
- 尝试把项目从算法跑通推进到可解释、可复盘的完整叙事

### 难点

- 真实时序数据不整洁，噪声、尺度和结构差异都很明显
- 异常检测不只是把分数跑出来，还要思考结果怎样落回任务语境
- 训练、评估和结果解释之间需要反复平衡

### 我从里面学到的

这个项目让我更清楚自己为什么会被算法问题吸引。我在意的不只是模型能不能收敛，还在意它有没有真的理解问题本身。

### GitHub

- <https://github.com/jolene718/MSCRED-NASA-Anomaly-Detection>

## 02. Curiosit-Atelier

### 为什么做

这个名字本身就很像我会做的东西。它更像一个把“好奇心”做成空间的仓库，不只是完成某个单点功能，而是试着把内容、视觉和体验感织到一起。

### 我做了什么

- 把一个更偏叙事感、展陈感的网页想法落成项目结构
- 关注页面气质、浏览节奏和内容组织方式
- 尝试让技术实现服务于表达，而不是喧宾夺主

### 难点

- 这类项目最难的往往不是把页面写出来，而是把气质写出来
- 内容、视觉和交互如果没有统一方向，就会很快变成一堆漂亮但空心的片段

### 我从里面学到的

我越来越确定，自己想做的东西不只是“能跑”，还应该有叙事感、有审美判断，也有自己想说的话。

### GitHub

- <https://github.com/jolene718/Curiosit-Atelier>

## 03. FogWar_Gobang

### 为什么做

这是一个一看名字就很有画面感的小作品。五子棋本身是熟悉规则上的再设计，而“战争迷雾”这种设定，会让一个传统游戏多出策略层和信息不对称带来的紧张感。

### 我做了什么

- 围绕五子棋玩法做带机制变化的设计尝试
- 把“看不见全部局面”的限制变成博弈体验的一部分
- 让项目从单纯功能实现转向更完整的玩法表达

### 难点

- 规则一旦变化，平衡性和可理解性就会一起变难
- 好玩的机制不等于好理解的机制，设计和实现需要同时照顾

### 我从里面学到的

我很喜欢这种带一点点实验性质的小项目，它提醒我，产品与系统设计很多时候都从“规则如何改变体验”开始。

### GitHub

- <https://github.com/jolene718/FogWar_Gobang>

## 04. Shapeville

### 为什么做

从名字来看，它更像一个偏视觉、偏交互、偏构成实验的项目。它吸引我的地方，是把抽象概念做得可感知，让人不是在看一堆功能按钮，而是在进入一个有一点性格的小世界。

### 我做了什么

- 尝试把形状、结构和交互体验组合成一个完整表达
- 让视觉语言本身承担一部分叙事功能
- 在实现层面练习把抽象想法变成可浏览的界面

### 难点

- 抽象题材很容易空，必须靠结构和交互把它落稳
- 视觉表达越轻，背后越需要清楚的取舍

### 我从里面学到的

我会反复被这种“形式感很强，但不能只有形式”的项目吸引，因为它们逼着我同时思考设计、叙事和实现。

### GitHub

- <https://github.com/jolene718/Shapeville>

## 05. loom-millennium-visualization

### 为什么做

这是一个明显偏可视化方向的项目。对我来说，可视化不是把数据画出来那么简单，它更像把原本不容易被看见的关系、结构和节奏重新组织给人看。

### 我做了什么

- 以可视化为核心表达方式组织信息
- 尝试让“观看”成为理解的一部分
- 在内容结构和视觉层之间建立更清楚的对应关系

### 难点

- 可视化项目很容易在“信息很多”和“真正清楚”之间失衡
- 不是元素越多越有效，重要的是节奏、重点和理解成本

### 我从里面学到的

我越来越在意信息如何被呈现，因为表达方式本身会改变一个内容被理解的方式。

### GitHub

- <https://github.com/jolene718/loom-millennium-visualization>

## 06. LJculture

### 为什么做

这个项目更接近“带现实场景的问题”，而不是纯练习仓库。它很像一个内容、文化或校园语境下的应用尝试，吸引我的地方在于：它要求技术不只是存在于本地运行，而是进入真实使用情境。

### 我做了什么

- 尝试把更具体的主题内容做成可使用的产品形态
- 关注信息组织、页面流程和真实用户视角
- 让内容表达和功能落地不要互相打架

### 难点

- 一旦进入真实语境，需求会比“做一个页面”复杂得多
- 内容、交互和实现必须同时被照顾，缺一块都会影响体验

### 我从里面学到的

我很在意项目是不是只是“做完”，还是有机会真的被使用、被理解、被留下来。

### GitHub

- <https://github.com/jolene718/LJculture>

## 07. LJweapp-images

### 为什么做

有些仓库不是主项目本体，但它们是项目真正能被组织起来的一部分。这个仓库更像资源层、素材层的支撑，它说明一个产品真正落地时，远不只有主代码仓那么简单。

### 我做了什么

- 为微信小程序相关项目整理和托管图像资源
- 让前端呈现与内容资源之间的协作更稳定
- 把零散素材变成项目可以复用的资产

### 难点

- 资源仓库看起来不起眼，但一旦失去秩序，项目维护会变得很痛苦
- 真正做产品时，“整理能力”并不比“开发能力”次要

### 我从里面学到的

项目的完成度，常常体现在这些不那么显眼的支撑层里。

### GitHub

- <https://github.com/jolene718/LJweapp-images>

## 08. HCI / mymllib

### 它们为什么也留在这里

并不是每个公开仓库都已经长成成熟作品。有一些更像课程留痕、实验台或者以后还会继续扩写的草稿仓库。`HCI` 和 `mymllib` 目前更接近这样的存在。

### 我希望它们以后长成什么

- `HCI`：更完整的人机交互记录、设计思路或实验过程
- `mymllib`：更清楚的机器学习方法整理或代码积累

我不想把这种“尚未长成”的阶段藏起来。对我来说，公开作品不只包含完成品，也包含还在生成中的方向感。

### GitHub

- <https://github.com/jolene718/HCI>
- <https://github.com/jolene718/mymllib>
